## Two-canonicalization pattern (per chopmob-cloud on A2A #1829, 2026-05-16)

The substrate boundary at the wire / structured layers is a **two-canonicalization pattern, not a one-canonicalization pattern**. Both are required; neither subsumes the other.

| Layer | Canonicalization | What it commits to | Why it's needed |
|---|---|---|---|
| **Wire** (RFC 9421 + RFC 9530 content-digest) | RFC 9530 digest over wire bytes | "this body at this moment crossed this HTTP boundary intact" | Survives proxies rewriting irrelevant headers, gzip/chunked encoding, TLS re-terminations |
| **Structured** (RFC 8785 JCS) | JCS over structured data fields | "this payload's semantic content hashes deterministically across implementations" | Cross-implementation hash equality (multi-language verifier byte-match) |

The non-overlap: content-digest survives transport mutations but doesn't survive payload re-encoding (a verifier that round-trips JSON through a different parser sees different wire bytes); JCS gives deterministic structured-data hashing but doesn't survive transport mutations (gzip changes wire bytes; JCS-canonical bytes don't change).

**Layering implication for v0.3.3:**
- L1 (wire signature, #1829) uses content-digest over wire bytes — transport integrity
- L3/L4 (CTEF claims, APS receipts) use JCS over structured data — payload semantics
- If a claim envelope at L3/L4 references the L1 signature tuple `(@method, @path, content-digest, created, nonce, keyid)`, the claim envelope's JCS over those tuple fields produces a deterministic cross-implementation hash, while the underlying content-digest in the tuple anchors back to the actual wire bytes the verifier saw. **Two-canonicalization stack: wire-bytes-digest at L1, JCS-over-structured-fields at L3/L4.**

Aligns with the x402-foundation/x402#2322 separation: per-row `content_hash = SHA-256(JCS(canonical fields))` for the structured layer + HMAC-SHA-256 over JCS-canonicalised envelope wrapping the per-row hashes for the bundle layer. Same two-layer pattern across four evidence classes (behavioral / regulatory / cryptographic / observational).

Reference: chopmob-cloud comment on a2aproject/A2A#1829 (2026-05-16). Production validation via AlgoVoi proxy-chain (Cloudflare → nginx → FastAPI) demonstrates the wire-bytes-digest property survives multi-hop transport mutations; the JCS-over-structured-fields property holds via 10 byte-match validated implementations in §3.8.

## Contribution rules

1. **No spec changes** in this working doc — it's a coordination surface, not the canonical spec. Spec changes land in `docs/internal/rfc-evidence-format-v1.md` (CTEF main RFC) once consensus is reached here.
2. **PRs welcome** for: filling TBD cells in the fixture matrix, drafting envelope-shape diff sub-sections, adding per-spec error codes to the unified enum draft.
3. **Closed claim_type set stays closed** — v0.3.3 does not expand `{identity, transport, authority, continuity}`. New attestation domains use `claim_subtype` inside the existing types.
4. **JCS canonicalization (RFC 8785) is non-negotiable.** Any v0.3.3 envelope change MUST produce byte-identical output across all 8 byte-match validated implementations.
5. **Negative-path coverage is mandatory** — every spec addition lands with at least one `expected_error_code` conformance vector. Fail-closed by construction.
