# pre-execution-verdict-v0 — the `verifier_attestation` envelope (v0.4)

The signed admission object the verifier produces and the gateway checks **before**
minting/reserving. Synthesizes: evidai's preflight seam, @haroldmalikfrimpong-ops's
binding_digest + admission shape, @azender1's exactly-once guard (`action_ref`),
@aeoess's verification-source provenance (`key_source`, #1829). Optional everywhere —
absent ⇒ un-gated, backward-compatible both ways.

## Shape

```json
{
  "envelope": "pre-execution-verdict-v0",
  "verifier_did": "did:web:trust.example.com",
  "subject_did": "did:web:example.com:agent:<id>",
  "admission": {
    "verdict": "admit | deny | flag",
    "dynamic_limit_usd": 50.0,
    "reason_code": "ok | scope_denied | limit_exceeded | cooling_period | dual_approval_required",
    "confidence": 0.0
  },
  "binding": {
    "charge_ref": "<gateway charge / idempotency key>",
    "amount_usd": 50.0,
    "nonce": "<unique-per-charge>",
    "binding_digest": "sha256(JCS({charge_ref, amount_usd, subject_did, nonce}))",
    "action_ref": "<content-addressed action_ref, per #1850 — names which preimage>"
  },
  "gates_rechecked": ["identity", "certificate", "key_lifecycle"],
  "key_source": "inline | cache | resolver",
  "issued_at": "2026-06-09T20:00:00Z",
  "expires_at": "2026-06-09T20:00:35Z",
  "freshness_ttl_seconds": 35
}
```
Signed: EdDSA over the JCS-RFC8785 canonical form; verifiable offline against
`verifier_did`'s JWKS.

## What the gateway does at preflight (pre-mint)
1. Verify the JWS against `verifier_did`'s published JWKS (no signer-controlled fetch on the hot path — see `key_source`).
2. Recompute `binding_digest = sha256(JCS({charge_ref, amount_usd, subject_did, nonce}))` and confirm it matches **this** charge — an attestation can't be replayed against a different charge.
3. Require `admission.verdict == "admit"` (else REFER/DENY per `reason_code`).
4. Reserve with `effective_budget = min(static_cap, admission.dynamic_limit_usd)`.
5. Confirm `expires_at` not passed (the ~30-60s window between verdict and charge).
6. `binding.action_ref` is the key for @azender1's exactly-once guard (PENDING → COMMITTED; retry returns the original receipt) — same preimage on both sides, independently verifiable.

## Field notes
- **`key_source`** (@aeoess, #1829): records whether the verifying key was `inline`, `cache`, or `resolver` — travels in the *signed* object, not just verifier logs, so a downstream auditor of the receipt knows whether verification depended on a signer-controlled fetch. Same signature, different trust posture.
- **`gates_rechecked`**: only the *fast* gates re-run at mint time; the slow evidence bundle doesn't move in the 30-60s window.
- **`reason_code`**: fixed vocabulary so DENY/FLAG is machine-actionable, not a freeform string.
- **Composition**: this is the proposed third part of the seam — **signed admission (this) → static cap + settlement + `charge_ref` (evidai, live) → exactly-once guard keyed on `action_ref` (@azender1, live + verified)**.

## Conformance
Implementers SHOULD publish a byte-verifiable fixture to the same bar as Row 8:
the signed object + JCS canonical bytes + SHA-256, the four scenarios
(admit / scope-deny / limit-deny / dual-approval-flag), verifiable on a fresh
clone against the verifier's published key. AgentGraph will publish the
verifier-side reference once the field names settle in-thread.
