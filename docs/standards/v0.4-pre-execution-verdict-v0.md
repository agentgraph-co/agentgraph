# pre-execution-verdict-v0 — the `verifier_attestation` envelope (v0.4)

The signed admission object the verifier produces and the gateway checks **before**
minting/reserving. Synthesizes: evidai's preflight seam, @haroldmalikfrimpong-ops's
binding_digest + admission shape, @azender1's exactly-once guard (`action_ref`),
@aeoess's verification-source provenance (`key_source`, #1829). Optional everywhere —
absent ⇒ un-gated, backward-compatible both ways.

## Shape

```json
{
  "envelope": "pre-execution-verdict-v0",
  "verifier_did": "did:web:trust.example.com",
  "subject_did": "did:web:example.com:agent:<id>",
  "admission": {
    "verdict": "admit | deny | flag",
    "dynamic_limit_usd": 50.0,
    "reason_code": "ok | scope_denied | limit_exceeded | cooling_period | dual_approval_required",
    "confidence": 0.0
  },
  "binding": {
    "charge_ref": "<gateway charge / idempotency key>",
    "amount_usd": 50.0,
    "nonce": "<unique-per-charge>",
    "binding_digest": "sha256(JCS({charge_ref, amount_usd, subject_did, nonce}))",
    "action_ref": "<content-addressed action_ref, per #1850 — names which preimage>",
    "action_ref_method": "argentum-core/action-ref-v1 | aps/draft-pidlisnyi-aps-01"
  },
  "gates_rechecked": ["identity", "certificate", "key_lifecycle"],
  "key_source": "inline | cache | resolver",
  "issued_at": "2026-06-09T20:00:00Z",
  "expires_at": "2026-06-09T20:00:35Z",
  "freshness_ttl_seconds": 35
}
```
Signed: EdDSA over the JCS-RFC8785 canonical form; verifiable offline against
`verifier_did`'s JWKS.

## What the gateway does at preflight (pre-mint)
1. Verify the JWS against `verifier_did`'s published JWKS (no signer-controlled fetch on the hot path — see `key_source`).
2. Recompute `binding_digest = sha256(JCS({charge_ref, amount_usd, subject_did, nonce}))` and confirm it matches **this** charge — an attestation can't be replayed against a different charge.
3. Require `admission.verdict == "admit"` (else REFER/DENY per `reason_code`).
4. Reserve with `effective_budget = min(static_cap, admission.dynamic_limit_usd)`.
5. Confirm the reserve will complete **strictly before** `expires_at` — a hard deadline, not advisory (see Normative requirements). If it cannot, re-verify or fail closed.
6. `binding.action_ref` is the key for @azender1's exactly-once guard (PENDING → COMMITTED; retry returns the original receipt) — same preimage on both sides, independently verifiable.

## Normative requirements
Resolving open items 2 (freshness) and 4 (replay) raised by @evidai on #1920.

### Freshness — `expires_at` is binding, not advisory (item 2)
The gateway MUST complete the reserve strictly before `expires_at`. `expires_at` is a
hard deadline on the **preflight-to-reserve** window, not merely a TTL on the verdict: if
the reserve would commit at or after `expires_at` (e.g. under load), the gateway MUST
re-verify (obtain a fresh attestation) or fail closed — it MUST NOT reserve against an
expired verdict. Verifiers SHOULD set `freshness_ttl_seconds` to bound the worst-case
preflight-to-reserve latency of conforming gateways (RECOMMENDED ≤ 60s), with
`expires_at = issued_at + freshness_ttl_seconds`. A gateway that cannot guarantee reserve
within the window MUST treat the attestation as expired. This makes `expires_at` binding
rather than advisory.

### Replay — `nonce` and the exactly-once guard are both REQUIRED (item 4)
Replay safety has two distinct surfaces; conforming gateways MUST implement both:
- **Cross-charge replay** is closed by `nonce` — it is in the `binding_digest` JCS preimage,
  so the attestation recomputes to a different digest against any other charge and is
  rejected at preflight step 2.
- **Same-charge retry replay** is closed by the exactly-once guard keyed on `action_ref`
  (PENDING → COMMITTED → SKIP, @azender1) — a retry of the same charge returns the original
  receipt rather than re-executing.
A normative `nonce` field *without* the exactly-once guard does NOT provide full replay
safety, and the guard without `nonce` does not bound cross-charge reuse. Neither alone is
sufficient.

## Field notes
- **`key_source`** (@aeoess, #1829): records whether the verifying key was `inline`, `cache`, or `resolver` — travels in the *signed* object, not just verifier logs, so a downstream auditor of the receipt knows whether verification depended on a signer-controlled fetch. Same signature, different trust posture. **`cache` is a derivation, not a place** (per @aeoess, #1829, 2026-06-09): a cached key inherits the trust posture of whatever populated it, so `cache` is defined as "previously verified via `inline` or `resolver` and pinned, with the population event (source + timestamp) auditable." This keeps the enum at three and closes the laundering path where a hot `resolver` fetch becomes `cache` one call later. Population provenance travels with the signed evidence (cf. @Liuyanfeng1234's `composition_ref`). The converged three-profile model (#1829): `resolver` proves via allowlist + fetch evidence; `cache` via a parent population event; `inline` via origin attestation (`key_origin_proof` + `key_origin_type`, @giskard09's composition-ref-v1). The same RFC 9421 signature passes all three — the receipt evidence is what distinguishes the trust posture.
- **`action_ref_method`** (@azender1, #1920): names which preimage derivation produced `action_ref` (argentum-core `action-ref-v1` vs APS `draft-pidlisnyi-aps-01`) so a downstream auditor knows which recipe to recompute. Same principle as `key_source` — the recompute recipe travels in the signed object, not inferred from the ref's shape.
- **`gates_rechecked`**: only the *fast* gates re-run at mint time; the slow evidence bundle doesn't move in the 30-60s window.
- **`reason_code`**: fixed vocabulary so DENY/FLAG is machine-actionable, not a freeform string.
- **Composition**: this is the proposed third part of the seam — **signed admission (this) → static cap + settlement + `charge_ref` (evidai, live) → exactly-once guard keyed on `action_ref` (@azender1, live + verified)**.

## Conformance
Implementers SHOULD publish a byte-verifiable fixture to the same bar as Row 8:
the signed object + JCS canonical bytes + SHA-256, the four scenarios
(admit / scope-deny / limit-deny / dual-approval-flag), verifiable on a fresh
clone against the verifier's published key. **AgentGraph's verifier-side reference is
published** at [`docs/conformance/verifier-attestation-v0/`](../conformance/verifier-attestation-v0/)
(`node verify_fixture.mjs` → 34/34; `cross_impl_check.mjs` → 11/11 byte-match vs
@haroldmalikfrimpong-ops's verifier impl + @evidai's gateway `gated-preflight-v1`).
`action_ref` follows the canonical JCS derivation of draft-giskard-aeoess-action-ref §3.
